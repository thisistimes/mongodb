⚠ ข้อควรระวัง
Date vs String: เก็บวันที่เป็น String "2024-01-15" จะเรียงลำดับผิด — ใช้ ISODate เสมอเพื่อให้ query ช่วงวันที่ถูกต้อง
Double vs Decimal128: 0.1 + 0.2 = 0.30000000000000004 ใน Double — ข้อมูลการเงินต้องใช้ Decimal128 เสมอ
Null vs field หาย: {"expiryDate": null} ต่างจาก document ที่ไม่มี field expiryDate — query {expiryDate: null} จับได้ทั้งสองแบบ
ObjectId เก็บอะไรบ้าง? 12 bytes: Timestamp 4 bytes (รู้เวลาสร้าง) + Machine ID 5 bytes + Counter 3 bytes — แปลว่าอ่านเวลาสร้าง document จาก _id ได้เลย!
Schema Design — Embedding vs Referencing หัวใจของการออกแบบ MongoDB ที่ดี
🏗️
ใน SQL เราคิดถึง table ก่อน แล้วค่อย JOIN ทีหลัง
ใน MongoDB เราคิดถึง access pattern ก่อน — ข้อมูลที่ query พร้อมกันควรอยู่ใน document เดียวกัน
📦 Embedding (Denormalized)
ซ้อนข้อมูลที่เกี่ยวข้องไว้ใน document เดียว
{ "_id": "con001", "status": "granted", "subject": { // ← embed "name": "สมชาย", "email": "s@ex.com" }, "purpose": { // ← embed "name": "การตลาด" } }
✓ อ่าน 1 query ได้ข้อมูลครบ
✓ เร็วมาก ไม่ต้อง JOIN
✗ ถ้า subject เปลี่ยนชื่อ ต้อง update ทุก consent
✗ document อาจใหญ่เกิน 16MB
🔗 Referencing (Normalized)
เก็บแค่ _id แล้วใช้ $lookup หรือ app join
// consents { "_id": "con001", "status": "granted", "subjectId": "sub001", // ← ref "purposeId": "pur001" // ← ref } // data_subjects { "_id": "sub001", "name": "สมชาย" }
✓ ข้อมูลไม่ซ้ำ แก้ที่เดียวได้เลย
✓ document เล็ก ประหยัด storage
✗ ต้อง 2 query หรือใช้ $lookup
✗ $lookup ช้ากว่า embed มาก
เมื่อไหร่ใช้อะไร
เงื่อนไข Embed Reference
ความสัมพันธ์One-to-One, One-to-FewOne-to-Many, Many-to-Many
ข้อมูลเปลี่ยนบ่อย?ไม่ค่อยเปลี่ยน (เช่น proof hash)เปลี่ยนบ่อย (เช่น ชื่อ, email)
query พร้อมกัน?ใช้ด้วยกันเสมอบางครั้งใช้แยกกัน
ขนาดข้อมูลย่อยไม่เกิน ~20 รายการจำนวนมาก/ไม่จำกัด
ตัวอย่างจริงproof, metadata, addresssubjectId, purposeId, logs
⚠ Anti-patterns ที่พบบ่อย
Unbounded Array: เก็บ logs ไว้ใน array ของ consent document — array โตไม่หยุด document เกิน 16MB
✓ แก้โดย: แยก consent_logs ออกเป็น collection แล้ว reference
Massive Document: embed ทุกอย่างจน field หลักร้อย — index ไม่มีประสิทธิภาพ
✓ แก้โดย: แยก field ที่ใช้บ่อยไว้ document หลัก ส่วนข้อมูลเสริมแยก collection
Polymorphic Hell: ใช้ collection เดียวเก็บข้อมูลต่างกันมากเกิน จน field overlap
✓ แก้โดย: แยก collection ตาม type หรือใช้ discriminator field + schema validation
$lookup — ดึง Reference ข้าม Collection
// Join consents กับ data_subjects db.consents.aggregate([ { $lookup: { from: "data_subjects", localField: "subjectId", foreignField: "_id", as: "subjectInfo" }}, { $unwind: "$subjectInfo" }, { $project: { status: 1, "subjectInfo.name": 1 }} ])
⚠ $lookup ช้ากว่า embed มาก — ใช้เฉพาะเมื่อ reference เป็น pattern ที่เหมาะสม ถ้า query ทำบ่อยควรพิจารณา denormalize
Transaction ใน MongoDB ACID multi-document transactions (v4.0+)
⚛️
MongoDB v4.0+ รองรับ multi-document ACID transaction ทำงานบน Replica Set
v4.2+ รองรับ transaction ข้าม Shard (distributed transaction)
แต่ถ้าออกแบบ schema ดี (embed ข้อมูลที่ต้อง atomic ไว้ด้วยกัน) มักไม่ต้องใช้ transaction เลย
ACID คืออะไร?
A — Atomicity
ทำทั้งหมดหรือไม่ทำเลย ถ้า error rollback ทั้งหมด
C — Consistency
ข้อมูลต้องถูกต้องตาม rule ก่อนและหลัง transaction
I — Isolation
transaction ที่ยังไม่ commit ไม่ส่งผลต่อ session อื่น
D — Durability
เมื่อ commit แล้วข้อมูลไม่หายแม้ server crash
ตัวอย่าง: Transaction ระบบ Consent (Node.js)
// withdraw consent พร้อมบันทึก log แบบ atomic // ถ้า step ใดล้มเหลว ทั้งคู่จะ rollback อัตโนมัติ const session = await client.startSession(); try { session.startTransaction({ readConcern: { level: 'snapshot' }, writeConcern: { w: 'majority' } }); // Step 1: อัปเดต consent status await db.collection('consents').updateOne( { _id: 'con001' }, { $set: { status: 'withdrawn', withdrawDate: new Date() } }, { session } ); // Step 2: บันทึก log await db.collection('consent_logs').insertOne( { consentId: 'con001', action: 'withdraw', actionDate: new Date() }, { session } ); await session.commitTransaction(); // ✓ บันทึกทั้งคู่ } catch (err) { await session.abortTransaction(); // ✗ ยกเลิกทั้งคู่ throw err; } finally { await session.endSession(); }
✓ เมื่อไหร่ควรใช้: ต้องอัปเดต 2 collection พร้อมกันแบบ atomic เช่น consents + consent_logs ถ้า step ใดล้มเหลวต้อง rollback ทั้งหมด
⚠ Transaction มี overhead: ช้ากว่า single document operation — ใช้เฉพาะเมื่อจำเป็น ถ้า embed ข้อมูลในที่เดียวได้ไม่ต้องใช้
✗ ข้อจำกัด: ต้องใช้บน Replica Set เท่านั้น | timeout default 60 วินาที | สร้าง collection ใหม่ใน transaction ไม่ได้
Single Document = ฟรี ACID: MongoDB มี atomic operation ในระดับ single document อยู่แล้วโดยไม่ต้องใช้ transaction — นี่คือเหตุผลที่ embed ดีกว่า reference ในหลายกรณี

MongoDB คืออะไร
MongoDB เป็น Document Database ประเภท NoSQL — แทนที่จะเก็บข้อมูลในตารางแบบ SQL MongoDB เก็บเป็น JSON document แต่ละชิ้น ซึ่งมีโครงสร้างอิสระจากกัน

หนึ่ง Document ≈ หนึ่งแถวใน SQL แต่สามารถซ้อน object และ array ข้างในได้เลย ไม่ต้อง JOIN ไปดึงตารางอื่น
{ "_id": "con001", "subjectName": "สมชาย ใจดี", "status": "granted", "purpose": { // ← object ซ้อนได้ "name": "การตลาด", "legal": "consent" }, "dataTypes": ["email", "name"] // ← array ได้เลย }
ลักษณะเด่น
◆ Schema ยืดหยุ่น — เพิ่ม field ใหม่ได้ทันที ไม่ต้อง migrate ทั้งตาราง
◆ Document-based — ข้อมูลที่เกี่ยวกันอยู่ใน document เดียว ไม่ต้อง JOIN
◆ Scale แนวนอน — กระจาย server ได้ง่ายด้วย Sharding
◆ Query ทรงพลัง — filter, sort, aggregate, text search ในที่เดียว
◆ เก็บข้อมูลเป็น BSON — อ่านเร็ว รองรับ type พิเศษเช่น Date, ObjectId
โครงสร้างหลัก
Database → เทียบได้กับ database ใน SQL
Collection → เทียบได้กับ table ใน SQL
Document → เทียบได้กับ row ใน SQL
Field → เทียบได้กับ column ใน SQL

โครงสร้างข้อมูลเปลี่ยนบ่อย เช่น product catalog ที่สินค้าแต่ละประเภทมี attribute ต่างกัน หรือ consent ที่มี field เพิ่มตาม version
✓ ใช้
ข้อมูลที่ดึงมาใช้พร้อมกัน เช่น consent + subject + purpose ควรอยู่ใน document เดียวเพื่อ read เร็ว
✓ ใช้
ข้อมูลปริมาณมากและต้องการ scale เช่น log, event stream, IoT — Sharding กระจาย server ได้ง่าย
✓ ใช้
MVP / Prototype — ไม่ต้องออกแบบ schema ล่วงหน้า เริ่ม dev ได้เลย ปรับ field ทีหลังได้
✓ ใช้
ข้อมูลที่มีโครงสร้างซ้อน เช่น user profile ที่มี addresses[], preferences{}, history[] — เก็บได้ใน document เดียวโดยไม่ต้องแตก 4 ตาราง
เมื่อไหร่ควรเลือก DB อื่น
✗ หลีกเลี่ยง
ธุรกรรมการเงิน ที่ต้องการ ACID เต็มรูปแบบและ relation ซับซ้อน — ใช้ PostgreSQL แทน
✗ หลีกเลี่ยง
ข้อมูลที่ต้อง JOIN เยอะมาก ทุก query เช่น ERP, accounting — SQL เหมาะกว่ามาก
✗ หลีกเลี่ยง
Full-text search ขั้นสูง เช่นต้องการ NLP, fuzzy, ranking — ใช้ Elasticsearch แทน
✗ หลีกเลี่ยง
Cache / Session ที่ต้องการความเร็วสูงสุด — ใช้ Redis ที่อยู่ใน RAM แทน
เปรียบเทียบกับ SQL ในงานเดียวกัน
สถานการณ์ SQL MongoDB
เพิ่ม field ใหม่ ALTER TABLE — กระทบทุก row เพิ่มใน document ใหม่ได้เลย
ดึง consent + subject SELECT ... JOIN 3 tables find() — document เดียวได้ครบ
เพิ่ม server ซับซ้อน ต้องอัพเกรดแนวตั้ง Sharding — เพิ่ม node ได้เลย
ข้อมูล consent version ต่างกัน ต้องมี nullable column เต็มไปหมด แต่ละ document มี field ต่างกันได้
ธุรกรรมการเงิน ACID เต็ม — เหมาะมาก รองรับ แต่ SQL ยังเหมาะกว่า
สรุป: MongoDB ไม่ได้ดีกว่า SQL ในทุกด้าน — แต่เหมาะกว่าเมื่อข้อมูลมี โครงสร้างซับซ้อน เปลี่ยนบ่อย หรือปริมาณมาก ระบบจริงหลายแห่งใช้ทั้งคู่ร่วมกัน เช่น MongoDB เก็บ consent และ profile + PostgreSQL เก็บข้อมูลการเงิน
